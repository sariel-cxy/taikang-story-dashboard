# GitHub 公开发布检查清单

这份清单用于把当前仪表盘发布到 GitHub Pages 之前做最后检查。

## 一次性判断

先确认这次发布属于哪一类：

- `公开发布`
  面向外部访问，默认任何人都可以看到页面和数据文件
- `半公开发布`
  链接可转发，但内容仍然需要人工审查
- `内部使用`
  不建议使用公开 GitHub Pages，除非数据已经脱敏

如果不是确定可公开，默认按 `不能公开` 处理。

## 核心风险点

当前站点是纯静态站点，以下内容一旦发布，访问者都能看到：

- `data/story-data.js` 中的全部故事数据
- 故事卡页中的详细字段
- 源链接、人物名称、备注等文本

这意味着不是只有页面上“显眼的内容”会公开，而是前端数据基本都可被访问。

## 发布前必查字段

发布前请重点检查 Excel 主表中的这些字段：

- `故事摘要`
- `关键人物名称`
- `金句`
- `背景`
- `核心情节`
- `关键转折/细节`
- `结果/影响`
- `价值意义/传播价值`
- `来源标题`
- `来源链接`
- `故事提供者`
- `备注`

这些字段最容易包含：

- 未公开口径
- 内部判断
- 外部不可见链接
- 人名、机构名、项目名
- 需要审批的传播内容

## 建议公开规则

建议按下面口径处理：

- `是否适合对外传播 = 公开可用`
  可以进入公开版站点候选池
- `是否适合对外传播 = 待内部审批 / 内部参考 / 敏感待审`
  默认不发布到公开站点

如果要做公开版 GitHub Pages，建议至少满足：

1. 只保留 `公开可用` 的故事
2. 去掉 `来源链接`
3. 去掉 `故事提供者`
4. 去掉 `备注`
5. 检查 `关键人物名称` 是否适合公开
6. 检查 `背景`、`核心情节` 中是否有内部信息

## 最短发布流程

1. 在 Excel 中筛出 `是否适合对外传播 = 公开可用`
2. 另存一份公开版 Excel
3. 必要时删除或清空这些列：
   - `来源链接`
   - `故事提供者`
   - `备注`
4. 运行导出脚本生成新的 `data/story-data.js`
5. 本地打开站点检查内容
6. 运行发布检查脚本
7. 再上传 GitHub

## 本地检查命令

```bash
python3 scripts/audit_publication.py
```

## 当前项目的现实情况

基于当前数据文件，已知至少存在：

- 外部链接
- `来源链接` 字段
- `关键人物名称` 字段
- `故事提供者` 字段
- `备注` 字段

这意味着当前版本不应直接默认公开发布。

## 建议做法

更稳妥的做法是维护两份数据：

- `工作版`
  用于内部补标签、维护全量故事
- `公开版`
  仅保留可公开字段和可公开故事

GitHub Pages 建议只发布 `公开版`。
